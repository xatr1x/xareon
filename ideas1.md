

# Запит на реалізацію функціоналу

Поточний проєкт вже має робочу основу (Tauri + Rust + TypeScript + SQLite) з CRUD для ігор. Реалізуйте наступні функції, зберігаючи існуючу архітектуру та дотримуючись усіх правил, описаних у AGENTS.md.

**ВАЖЛИВО:**
- Не порушуйте існуючу архітектуру.
- Вся бізнес-логіка належить бекенд-сервісам.
- Рівень репозиторію відповідає за доступ до бази даних.
- Після завершення реалізації оновіть AGENTS.md.

---

# 1. Ігровий журнал (Щоденник)

## Мета

Додаток насамперед є персональним ігровим журналом. Кожна гра повинна мати власний щоденник, у якому я можу періодично записувати свої думки під час проходження.

Це не коментарі. Це записи щоденника і вони повинні розглядатись як одна з основних сутностей додатку.

## Бекенд

Спроєктуйте та реалізуйте все необхідне:

- схему бази даних
- міграції
- доменну модель
- репозиторії
- сервіси
- команди
- валідацію

Кожен запис щоденника повинен містити щонайменше:

- id
- game_id
- дату/час створення
- дату/час останнього редагування
- текст

Архітектура повинна дозволяти легко розширювати записи у майбутньому (наприклад, додати скріншоти, настрій, теги чи інші метадані).

## Фронтенд / UX

Я свідомо не описую інтерфейс детально.

Спроєктуйте інтерфейс, який природно впишеться у існуючий додаток.

Щоденник повинен відчуватись як хронологія спогадів.

Мої очікування:

- відкриття гри має дозволяти перейти до її щоденника;
- читати записи має бути зручно;
- створення нового запису має бути швидким;
- редагування та видалення мають бути інтуїтивними;
- найновіші записи мають відображатись першими;
- дати/час повинні бути чітко видимими;
- інтерфейс має відповідати існуючій мінімалістичній темній темі.

Використовуйте власний розсуд для створення чистого та приємного UI.

# 2. Кілька жанрів

Гра може належати до кількох жанрів.

При створенні чи редагуванні гри я повинен мати змогу ввести:

Action, Adventure, RPG

Додаток має автоматично розпізнавати значення.

## База даних

НЕ зберігайте жанри як рядок, розділений комами.

Нормалізуйте базу даних правильно.

Спроєктуйте відповідну схему, щоб жанри стали багаторазовими сутностями, а ігри могли мати багато жанрів.

Архітектура повинна дозволяти у майбутньому додати:

- статистику за жанрами
- фільтрацію за жанрами
- керування жанрами

без переробки бази даних.

В UI можна залишити простий інпут із комами, якщо це забезпечує найкращий користувацький досвід.

# 3. Пошук, фільтрація та сортування

Поточний список ігор має перетворитись на потужний браузер.

Реалізуйте систему фільтрації, яка підтримує комбінації фільтрів.

Приклади:

- всі Action-ігри;
- всі Strategy-ігри;
- ігри з Action АБО RPG;
- ігри з Action І Adventure;
- ігри, завершені у певному році;
- ігри, розпочаті у певному році;
- ігри, у які грали у певному році;
- всі заплановані ігри;
- всі покинуті ігри;
- всі завершені ігри;
- ігри за платформою;
- ігри за діапазоном оцінки.

Бекенд повинен надати гнучкий API для запитів, а не багато вузько сфокусованих ендпоінтів.

Система фільтрації повинна легко розширюватись майбутніми фільтрами.

## Фронтенд

Спроєктуйте інтерфейс фільтрації самостійно.

Інтерфейс має залишатись мінімалістичним і не перевантажувати користувача.

Віддавайте перевагу згортаним фільтрам, чіпсам, дропдаунам чи іншим чистим десктопним патернам UI, де це доречно.

Сортування має підтримувати щонайменше:

- назву
- дату початку
- дату завершення
- рік релізу
- оцінку
- статус

Підтримуйте як зростаючий, так і спадний порядок.

# Загальні очікування

Під час реалізації цих функцій:

- зберігайте існуючу архітектуру;
- робіть код модульним;
- уникайте дублювання;
- використовуйте правильні практики Rust та TypeScript;
- підтримуйте мінімалістичний UI;
- забезпечуйте плавний та інтуїтивний UX;
- думайте про майбутню розширюваність до реалізації.

Якщо потрібне архітектурне рішення — обирайте те, що забезпечує найкращу довгострокову підтримку, а не найшвидшу реалізацію.

Після завершення обов'язково оновіть AGENTS.md, щоб він точно відображав усі архітектурні, базові та функціональні зміни, впроваджені цією реалізацією.
